# 线程池
## 定义
线程池是**维持**和**管理固定**线程的池式结构  
**维持** ：避免线程的频繁创建和销毁，从而减小系统开销  
**管理** ：管理线程状态 ，有任务则执行任务 ，无任务则休眠以腾出CPU  

## 线程池的组成
固定数目的线程+任务队列  
遵循**生产者--消费者**模型 

## 代码分析
### 线程池的构造
    1.创建 n 个工作线程，线程池启动后，所有工作线程进入休眠状态（通过条件变量等待）；
    2.当有新任务加入任务队列时，通过条件变量通知至少一个（或所有）休眠的工作线程，被唤醒的线程竞争互斥锁后从任务队列取出任务执行；
    3.当收到 “停止线程池” 的指令时，线程池不再接收新任务，所有工作线程在唤醒后，执行完任务队列中剩余的所有任务，且不再进入休眠等待，最终正常退出。

### 添加任务
    1.上锁
    2.判断线程池状态
    3.添加任务到任务队列

### 线程池的析构
    1.终止变量设置为true
    2.notify_all()唤醒所有正在该条件变量上等待的线程
    3.阻塞等待线程执行完毕+触发系统回收线程资源

## 与线程池相关的一些问题
### 1.为什么要固定线程数量？
    线程数量继续增加，由于系统资源的限制，不再带来性能的提升，反而带来负担

### 2.lock_guard和unique_lock的区别？
    两者都是c++中用于RAII方式管理互斥锁的包装器；
    其中unique_lock配合条件变量，支持中途解锁，从而灵活控制锁；
    unique_lock可移动，即将一个unique_lock对象对锁的管理权，通过std::move转移给另一个unique_lock对象，但两者都不可复制；
    unique_lock内部有更多的状态，从而会有更大的开销，不过这个开销属于“可接受的轻量级成本”。

### 3.析构函数中为什么使用join()而不是detach()?
    线程池析构时需要「可控地等待所有工作线程安全退出」，确保剩余任务执行完毕、资源不泄漏、避免野线程访问已销毁的对象；而 detach()后会直接释放线程池的核心成员(如mutex_,condition_等)，从而失去对线程的控制，可能会造成段错误或者任务丢失。
